# エージェント指示書

あなたはソフトウェアプロジェクトで作業する自律型コーディングエージェントです。

## ワーカー記述ルール

ワーカーファイルの先頭にそのAPIの説明とリクエスト例、レスポンス例をコメントで必ず記述する。

## API設計規約

- すべてのエンドポイントは **POST メソッド固定**。REST的なHTTPメソッド使い分け（GET/PUT/DELETE）はしない
- 操作の種類はパス名で区別する（例: `/location-save/{key}`, `/location-get/{key}`）

## エラーレスポンス規約

- APIエラーレスポンスには `error` フィールド（人間が読める概要）に加え、デバッグに役立つ `details` フィールドを含める
- Schema デコードエラーは `ParseError.message` を `details` に含め、どのフィールドが不正かを伝える
- 非同期ジョブの例外エラー (`errorValue`) には `endpoint` フィールドでどのハンドラで失敗したかを示す
- Scriptable クライアントのエラー表示では、`error` と `details` を構造化して表示する（生JSONダンプは避ける）

## タスク

0. `progress.txt` の進捗ログを読む（最初に Codebase Patterns セクションを確認）。次にやるタスク領域をなんとなく決める。
1. git commit / git pushが必要か確認する。
2. CIの結果を確認する。完了していなければ待機する。
3. **【開始前確認】** テスト・カバレッジのベースラインを記録する
   ローカルではなくCIからカバレッジの結果を取得する
4. `tasks/*.md` を読む。**`tasks/inserted-tasks.md` が存在し未完了タスクがあれば、他のタスクより優先して対処すること。**
5. `task-processing.md` に実行するタスクを書き写す。どのファイルから書き写したか明記する。既に存在すれば進捗を確認し、続きから開始する。
   このとき、基本的にはワンラインのみを写す。その下に周辺情報としてそのタスク周辺の情報を書くことはできる。
   タスク一覧側には実行中 `[-]` マークへの書き換えを実施する。
   対象とするタスクは、サブタスクであっても一個までとし、それ以外はあくまでもコンテキストとして転機するのみとし、今回集中するタスクがどれであるかは明瞭にする。
   **書き写したら、実装に入る前に以下の計画を `task-processing.md` に記述する:**
   - **テスト計画:** どのテストを追加・更新するか。どのテストファイルに書くか。
   - **ストーリー計画:** UI変更がある場合、どのストーリーを追加・更新するか。
   - **バグ修正の場合:** 再現条件を明確にし、デグレを防ぐためのリグレッションテスト（そのバグを再現するテスト）を先に書く計画を立てる。
6. その単一のタスクを実装する。**ステップ4で立てたテスト・ストーリー計画に沿って進める。**
7. 品質チェック: **typecheck, lint, test はローカルで実行しない。git push して CI の結果で確認すること。**
8. 再利用可能なパターンを発見した場合は CLAUDE.md ファイルを更新する（下記参照）
9. すべての変更をコミット。メッセージは**日本語**で: `feat: [領域] - [ストーリーの内容を日本語で簡潔に]`
10. git push、CIの結果を確認
11. `tasks/*.md` を更新する（下記「タスクファイルのアーカイブ」参照）。
12. `task-processing.md` の中身を空にする。完全に終わっていなければ、申し送り事項を追記して終了する。
13. **【完了前確認】** テスト・カバレッジを再確認し、低下していないことを確認
    低下や懸念、修正可能箇所があれば妥協なく修正する
14. `progress.txt` に進捗を追記する（カバレッジの変化も記録）

## タスクファイルのアーカイブ

タスク完了後、`tasks/*.md` の整理を以下のルールで行う:

### 個別タスク項目の完了

- 完了した項目は `[x]` に書き換える。
  - 特筆すべき選択や簡潔な対応方針の決定などを書き加える

### 完了項目のアーカイブ（部分的移動）

- **ファイル内に `[x]` が溜まってきたら、完了済み項目を `tasks-archived/` に移す。**
- ファイル全体が完了していなくても、**完了済みセクション・完了済み項目群は部分的にアーカイブしてよい。**
- アーカイブ先: `tasks-archived/<元のファイル名>` に追記（存在しなければ新規作成）。
- ただし、不完全に完了したもの、ゴールを完全に達成しきらなかったものは `tasks-retired/...` に移動する
  - ただ途中で終わらせただけであれば、次回に持ち越すのみで、これは利用しない。
- 元ファイルからは `[x]` 項目を削除し、未完了項目だけを残す。
- これにより、タスクファイルは常に**未完了の作業だけ**が見える状態を維持する。

### タスクファイル全体の完了

- ファイル内のすべてのタスクが完了したら、ファイル全体を `tasks-archived/` に `mv` する。
- 元の `tasks/` からは消える。
- **例外: `inserted-tasks.md` はファイルごとの `mv` はしない**（下記参照）。

### `inserted-tasks.md` の特別ルール

`inserted-tasks.md` は随時追加される差し込みタスクの置き場であり、ファイル自体は常に `tasks/` に残す。

- 完了した項目は `tasks-archived/inserted-tasks.md` へ移動し、元ファイルからは削除する。
- ファイル全体の `mv` は**しない**。すべてのタスクが完了しても、空の状態で `tasks/` に残しておく。

### 例

```bash
# 部分的アーカイブ: 完了項目を移動
# 1. 完了済み項目を tasks-archived/inserted-tasks.md に追記
# 2. 元ファイルから完了済み項目を削除

# 通常のタスクファイル全体完了: ファイルごと移動
mv tasks/foo.md tasks-archived/foo.md

# inserted-tasks.md はファイルごとのmvはしない（項目の移動のみ）
```

## 進捗レポートのフォーマット

progress.txt に追記する（上書きせず、必ず追記）:

```
## [日時] - [領域]
- 実装した内容
- 変更したファイル
- **今後のイテレーションへの学び:**
  - 発見したパターン（例: 「このコードベースは Y に X を使用している」）
  - 遭遇した注意点（例: 「W を変更する際は Z の更新を忘れずに」）
  - 有用なコンテキスト（例: 「評価パネルはコンポーネント X にある」）
---
```

学びセクションは非常に重要 - 将来のイテレーションが同じ間違いを繰り返さず、コードベースをより良く理解するのに役立つ。

## パターンの集約

**再利用可能なパターン**を発見した場合、progress.txt の**先頭**にある `## Codebase Patterns` セクションに追加する（存在しない場合は作成）。このセクションには最も重要な学びを集約する:

```
## Codebase Patterns
- 例: 集計には `sql<number>` テンプレートを使用
- 例: マイグレーションには常に `IF NOT EXISTS` を使用
- 例: UI コンポーネント用に actions.ts から型をエクスポート
```

**汎用的で再利用可能な**パターンのみを追加し、ストーリー固有の詳細は含めない。

## CLAUDE.md ファイルの更新

コミット前に、編集したファイルに関連して近くの CLAUDE.md ファイルに保存すべき学びがないか確認する:

1. **編集したファイルのあるディレクトリを特定** - どのディレクトリを変更したか確認
2. **既存の CLAUDE.md を確認** - それらのディレクトリまたは親ディレクトリに CLAUDE.md があるか確認
3. **価値ある学びを追加** - 将来の開発者/エージェントが知るべきことを発見した場合:
   - そのモジュール固有の API パターンや規約
   - 注意点や非自明な要件
   - ファイル間の依存関係
   - その領域のテストアプローチ
   - 設定や環境要件

**良い CLAUDE.md 追加の例:**

- 「X を変更する際は、同期を保つために Y も更新すること」
- 「このモジュールはすべての API 呼び出しにパターン Z を使用」
- 「テストには PORT 3000 で dev サーバーが起動している必要がある」
- 「フィールド名はテンプレートと完全に一致する必要がある」

**追加しないもの:**

- ストーリー固有の実装詳細
- 一時的なデバッグメモ
- progress.txt に既にある情報

そのディレクトリでの将来の作業に役立つ**本当に再利用可能な知識**がある場合のみ CLAUDE.md を更新する。

## テスト・カバレッジ確認（必須）

**100%カバレッジを目指す** - これはアーキテクチャの品質指標である。

progress.txt にベースラインカバレッジを記録する。

### カバレッジが低下した場合

**カバレッジ低下はレッドフラグ。** コミット前に必ず対処:

1. **新コードにテストを追加したか？** → テストを書く
2. **不純な領域（I/O、外部API等）か？** → 分離して除外設定に追加
3. **テストが書きづらいか？** → 設計を見直す（純粋と不純の分離）

**カバレッジ低下を放置して完了扱い、タスクの移動をしてはならない。**

### 不純な領域の分離

以下はカバレッジ対象外として分離・抽象化する:

- I/O処理（ファイル、ネットワーク、DB接続の実体）
- フレームワーク固有のエントリーポイント
- 環境依存の設定・初期化コード

これらは**インターフェース/型で抽象化**し、カバレッジ除外設定に追加する。

## Storybook ストーリーの品質要件

- ストーリーには基本的に **play 関数を書くこと**。ユーザー操作のシミュレーションや基本的なアサーションを含め、インタラクションテストとして機能させる
- play 関数がないストーリーは表示確認のみのため、意図的に静的表示だけを確認したい場合を除き避ける

## 品質要件

- すべてのコミットはプロジェクトの品質チェック（typecheck, lint, test）を通過すること
- **テストが通り、カバレッジが低下していないことを確認してからコミット**
- 壊れたコードをコミットしない
- 変更は集中的かつ最小限に
- 既存のコードパターンに従う

## 再発防止・変更漏れ防止

問題が起きたあと、またコードを変更するときは常に以下を意識する:

- **型エラー・チェックエラーとして検出できないか？** 変更が必要な箇所を見落とした場合、それをコンパイル時に検出できる仕組みにできないか考える
- **exhaustive check を活用する。** switch文やunion型の分岐では網羅性チェックを必ず入れ、変化に強制的に対応させる
- **プロパティ追加の検出:** オブジェクトのプロパティが増えたことに気付けなかった場合、`{ knownProp1, knownProp2, ...rest } = obj; assertRestExhaustive(rest);` パターンの利用を検討する
- **変更時に一緒にすべきことをコード内に明記する。** 関連するファイル・テスト・型定義など、一緒に変更すべきものをコメントやドキュメントで明示しておく（テストをどう更新すべきかも含め）
- **gitコミット差分を振り返る。** 問題が漏れた場合、差分を見て「この変更を最初からするにはどうすれば良かったか」を考え、仕組みとして再発を防ぐ

## Effect.ts 規約

本プロジェクトでは Effect.ts を段階的に採用している。以下の規約に従うこと。

### 使い分け

| 機能                                                      | 用途                                                                               | 使わない場面                       |
| --------------------------------------------------------- | ---------------------------------------------------------------------------------- | ---------------------------------- |
| `Schema.TaggedClass`                                      | ASTノード等のデータ型定義。`_tag` で discriminated union。シリアライゼーション対象 | エラー型（→ `Data.TaggedError`）   |
| `Data.TaggedError`                                        | エラー型。`_tag` を自動付与。`Either.left` / `Effect.fail` で生成                  | データ型（→ `Schema.TaggedClass`） |
| `Either`                                                  | Result型（`Right` = 成功, `Left` = 失敗）。公開APIの戻り値                         | 内部実装（→ `Effect.gen`）         |
| `Effect.gen`                                              | 複数のバリデーションチェーンがある関数。短絡評価が明示的になる                     | 単純な関数や単一の分岐のみの処理   |
| `Layer` + `Context.Tag`                                   | 副作用（localStorage, fetch等）の抽象化。テストでインメモリ実装に差替え            | 純粋関数のみのモジュール           |
| `Schema.decodeUnknownEither` / `Schema.encodeUnknownSync` | JSON import/export の型安全なシリアライゼーション                                  | 内部のデータ変換                   |

### パターン

#### 内部 Effect → 公開 Either ラッパー

```typescript
// 内部: Effect.gen で実装（"Effect" サフィックス）
const validateFooEffect = (...): Effect.Effect<Success, FooError> =>
  Effect.gen(function* () {
    // yield* Effect.fail(new SomeError({})) で短絡
  });

// 公開API: Either を返す（サフィックスなし）
export const validateFoo = (...): Either.Either<Success, FooError> =>
  Effect.runSync(Effect.either(validateFooEffect(...)));
```

#### DI（Layer + Context.Tag）

```typescript
// サービス定義
class StorageService extends Context.Tag("StorageService")<
  StorageService,
  { readonly getItem: (key: string) => Effect.Effect<string | null> }
>() {}

// 本番Layer
const BrowserStorageLayer = Layer.succeed(StorageService, { ... });

// テストLayer
const { layer } = createInMemoryStorageLayer();

// 使用: yield* ServiceName で取得
const storage = yield* StorageService;
```

#### エラー型定義

```typescript
// 空ペイロード: Record<string, never> を使う（{} は ESLint エラー）
export class FooMissing extends Data.TaggedError("FooMissing")<
  Record<string, never>
> {}
// ペイロードあり
export class FooParseError extends Data.TaggedError("FooParseError")<{
  readonly nodeId: string;
}> {}
// union型として集約
export type FooError = FooMissing | FooParseError;
```

### 注意事項

- `Schema.optionalWith({ as: "Option" })` は union schema で使えない。`Schema.optional` を使う
- `Either<A, E>` は `Right(A)` = 成功, `Left(E)` = 失敗（Effect.ts の慣例）
- UI層（React コンポーネント）では `Effect.runSync` / `Either.isRight` で境界を作る。UI層が Effect の内部型を直接参照しない
- `context7` の `resolve-library-id` → `query-docs` で最新APIを確認してから実装する
- 再帰型は `Schema.suspend((): Schema.Schema<T> => T)` で定義

## アーキテクチャ方針

とにかく独立した、そしてなるべく純粋な、よくテストされた部分をしっかり作り込んで、それを大量に並べて、それをちょっと結合しただけのものとしてアプリケーションが成立するように徹底的に意識せよ。

## 重要事項

- 1イテレーションにつき1ストーリーのみ作業
- **毎イテレーション必ずコミットする** - ストーリーが完了したら必ずコミット。コミットなしで終了しない
- 開始前に progress.txt の Codebase Patterns セクションを読む
